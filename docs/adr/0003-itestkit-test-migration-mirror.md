# ADR-0003: itestkit consumers maintain a plain-SQL migration mirror under tests/integration/testdata/

**Status:** accepted
**Date:** 2026-05-13

## Context

Every consumer of `itestkit` calls `StartStack` with `Postgres.ApplyMigrations: true` and points `MigrationPaths` at a directory of `.sql` files. itestkit executes each file with `db.Exec(script)` — it does not parse goose/migrate annotations. But several consumer repos (e.g. `onetrust-client-portal-api`) keep production migrations in goose format with `-- +goose Up` and `-- +goose Down` directives, where the Down section is in the same file as the Up. Passing those files to itestkit would execute both halves, undoing every migration.

Three reconciliations were considered:

- Rewrite every consumer's production migrations to plain SQL (drop goose entirely).
- Teach itestkit to parse goose/migrate annotations natively.
- Maintain a separate plain-SQL mirror under `tests/integration/testdata/postgres/<instance>/migrations/`, generated from the goose source by a Task target, with a CI check that fails on drift.

## Decision

Every consumer maintains a plain-SQL mirror at `tests/integration/testdata/postgres/<instance>/migrations/`. The mirror is generated by `task itest:sync-migrations`, which strips `-- +goose Up` lines and cuts everything from `-- +goose Down` onward, writing the result back under the same filename. CI runs the same target and fails when `git diff --exit-code tests/integration/testdata/` is non-empty.

## Why

- itestkit stays a thin Testcontainers wrapper. No migration-tool awareness, no new dependencies, no future support burden for tools we have not met yet.
- Each consumer picks its own production migration tool (goose, migrate, dbmate, plain psql). The test-time format is uniform regardless.
- Production migrations remain the single source of truth. The mirror is a generated artifact, treated like `*.pb.go`: committed but never hand-edited.
- The CI check rules out silent drift. A developer who forgets to run the sync target fails CI before the test schema diverges from production.

## Considered and rejected

- **Rewrite consumer migrations to plain SQL.** Forces a tool migration across every consumer repo, plus dev retraining and runbook updates, for the benefit of itestkit alone. Disruptive.
- **Teach itestkit goose/migrate.** Pulls a migration-tool dependency into the lib for one consumer's convenience, blocks future support for tools we have not met, and conflates "spin up infra" with "manage migrations" — two distinct concerns.
- **Run the consumer's migration tool inline from `Bootstrap`.** Every consumer would wire its tool against the DSN itestkit returned, and Bootstraps would diverge by toolchain. Defeats the goal that engineers moving between repos read the same surface.

## Consequences

- Every consumer commits `tests/integration/testdata/postgres/<instance>/migrations/`. It is generated but versioned, mirroring `*.pb.go` workflow.
- The mirror is read-only — direct edits are forbidden. Developers edit the goose source and re-run `task itest:sync-migrations`.
- The sync target's stripping rules (regex on `-- +goose Up`, cut at `-- +goose Down`) are part of the contract. Migration files MUST place the Up section above the Down section. Files that use goose `StatementBegin`/`StatementEnd` blocks or other directives that the simple awk splitter cannot handle need manual mirror authoring; the doc captures this escape hatch.
- Consumers using a migration tool that already produces plain `.sql` files (e.g. `golang-migrate` with separate `*.up.sql` / `*.down.sql` files) skip the mirror entirely and point `MigrationPaths` at the up-files directory directly. The mirror is goose-specific.
